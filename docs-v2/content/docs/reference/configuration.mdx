## Config file location

```
~/.opentabs/config.json
```

Override the config directory with the `OPENTABS_CONFIG_DIR` environment variable. The file is created automatically on first server run with `0600` permissions.

## Example config

```json
{
  "plugins": [
    "../plugins/slack",
    "/Users/you/projects/my-opentabs-plugin",
    "opentabs-plugin-jira"
  ],
  "tools": {
    "slack_delete_message": false
  },
  "secret": "a1b2c3d4e5f6...64 hex characters..."
}
```

## Field details

### plugins

A unified array of plugin specifiers. Local paths start with `./`, `../`, `/`, or `~/`; anything else is treated as an npm package name resolved via `node_modules`.

Local paths are resolved from the config directory (`~/.opentabs/` by default). Symlinks are resolved via `realpath()` and paths outside `$HOME` or `$TMPDIR` are rejected.

### tools

Disable specific tools without removing the plugin:

```json
{
  "tools": {
    "slack_delete_message": false,
    "slack_send_message": false
  }
}
```

Only include entries for tools you want to disable — absent keys default to enabled.

### secret

Used for authentication across all transports:

- **`/mcp`** and admin endpoints: `Authorization: Bearer <secret>` header
- **`/ws`**: `Sec-WebSocket-Protocol` header (keeps the secret out of URLs and logs)

Auto-generated on first server run. If missing from an existing config, the server generates a new one and writes it back.

## Dev vs production mode

The MCP server runs in **production mode** by default and optionally in **dev mode** for plugin developers.

### Production mode (default)

```bash
bun platform/mcp-server/dist/index.js
```

- Discovers plugins from the `plugins` array once at startup
- No file watchers — no background filesystem activity
- `POST /reload` returns 405 — restart the server to pick up plugin changes

### Dev mode

```bash
bun --hot platform/mcp-server/dist/index.js --dev
```

Or with the environment variable:

```bash
OPENTABS_DEV=1 bun --hot platform/mcp-server/dist/index.js
```

- **File watchers** monitor local plugin `dist/` directories — the server auto-reloads plugins when you rebuild them
- **Config watching** detects changes to `config.json` and re-discovers plugins
- **`POST /reload`** endpoint is available for manual reload triggers
- **`bun --hot`** enables hot reload — rebuilt server files are picked up without restarting the process

The `/health` endpoint includes a `mode` field (`"dev"` or `"production"`) so you can verify which mode the server is running in.

## Config directory structure

```
~/.opentabs/
├── config.json
├── extension/
│   ├── manifest.json
│   ├── background.js
│   ├── adapters/
│   │   ├── slack.js
│   │   └── ...
│   └── .opentabs-version
└── logs/
```

The `extension/` directory is managed by `opentabs setup` and `bun run build`. Load it as an unpacked extension in `chrome://extensions`.

Config writes use an atomic pattern: content is written to a temp file then renamed over the target. A mutex serializes concurrent writes.
