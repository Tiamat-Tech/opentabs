Each plugin in OpenTabs has a **tab state** that reflects whether the user has the target web app open and ready for tool execution. The extension computes state per-plugin and reports changes to the MCP server in real time.

## Three States

| State | Meaning | When |
| --- | --- | --- |
| `closed` | No matching tab exists | No Chrome tab matches the plugin's URL patterns |
| `unavailable` | Tab exists but not ready | A matching tab is open, but the adapter's `isReady()` returns `false` (e.g., user is not logged in, page still loading) |
| `ready` | Tab exists and ready | At least one matching tab has an injected adapter and `isReady()` returns `true` |

State is computed **per-plugin**, not per-tab. If a plugin has multiple matching tabs (e.g., two Slack windows), the plugin is `ready` as long as _any_ matching tab is ready.

## State Computation

The extension computes a plugin's state by:

1. **Finding matching tabs** — queries `chrome.tabs` for tabs whose URL matches the plugin's URL patterns (Chrome match patterns like `*://*.slack.com/*`)
2. **Probing readiness** — for each matching tab, executes `chrome.scripting.executeScript` in the MAIN world to call the adapter's `isReady()` function
3. **Determining state**:
   - No matching tabs → `closed`
   - Matching tabs exist, but none have `isReady() === true` → `unavailable`
   - At least one tab has `isReady() === true` → `ready`

The `isReady()` probe has a **5-second timeout**. If the adapter doesn't respond in time, the tab is treated as unavailable.

## Tab Ranking

When multiple tabs match a plugin's URL patterns, the extension ranks them to decide which tab executes a tool call:

| Rank | Condition | Example |
| --- | --- | --- |
| 3 (best) | Active tab in the focused window | The Slack tab you're looking at right now |
| 2 | Active tab in any window | A Slack tab that's active in a background window |
| 1 | Any tab in the focused window | A non-active Slack tab in the window you're working in |
| 0 | Any other matching tab | A Slack tab in a minimized window |

During tool dispatch, the extension tries tabs in rank order. If the highest-ranked tab fails (adapter not ready or tab inaccessible), it falls back to the next tab. This means keeping the target web app in an active, focused tab gives the best results.

## State Change Notifications

The extension sends state updates to the MCP server over WebSocket using two JSON-RPC 2.0 messages:

### `tab.syncAll`

Sent on WebSocket connect or reconnect. Contains the full state map for all plugins:

```json
{
  "method": "tab.syncAll",
  "params": {
    "tabs": {
      "slack": { "state": "ready", "tabId": 42, "url": "https://app.slack.com/..." },
      "github": { "state": "closed", "tabId": null, "url": null }
    }
  }
}
```

### `tab.stateChanged`

Sent when a single plugin's state changes:

```json
{
  "method": "tab.stateChanged",
  "params": {
    "plugin": "slack",
    "state": "ready",
    "tabId": 42,
    "url": "https://app.slack.com/..."
  }
}
```

The extension suppresses redundant notifications — if a tab event triggers a recomputation but the resulting state is the same as the cached state, no message is sent.

## When State Is Recomputed

State recomputation is triggered by three browser events:

| Event | Trigger | Scope |
| --- | --- | --- |
| URL change | `chrome.tabs.onUpdated` with `changeInfo.url` | Plugins matching the new URL + plugins with an active state for that tab |
| Page load complete | `chrome.tabs.onUpdated` with `changeInfo.status === 'complete'` | All plugins matching the loaded URL (adapters are injected first, then state is probed) |
| Tab closed | `chrome.tabs.onRemoved` | All plugins (the removed tab might have been the only matching tab) |

State computations for different plugins run in parallel, but computations for the _same_ plugin are serialized with a per-plugin lock to prevent cache corruption from concurrent events.

## How State Affects Tool Dispatch

When an AI agent calls a tool, the MCP server dispatches it to the extension. The extension's response depends on the plugin's tab state:

1. **`ready`** — the extension executes the tool on the best-ranked matching tab. If that tab's adapter reports it's not ready at dispatch time (race condition), the extension falls back to the next matching tab.

2. **`unavailable`** — the extension still attempts dispatch. The tool may succeed if the adapter became ready between the last state check and the dispatch, or it may fail with an adapter-not-ready error (code `-32002`).

3. **`closed`** — the extension returns an error immediately (code `-32001`, no matching tab). The MCP server wraps this in a user-friendly error explaining which web app needs to be open.

### Tool Dispatch Timeouts

| Timeout | Value | Purpose |
| --- | --- | --- |
| MCP server dispatch | 30 seconds | Maximum time the server waits for a tool result before timing out |
| Extension script execution | 25 seconds | Maximum time for `chrome.scripting.executeScript` — fires before the server timeout to guarantee a response |
| `isReady()` probe | 5 seconds | Maximum time to wait for a readiness check during state computation |

The extension timeout (25s) is deliberately shorter than the server timeout (30s) to ensure the extension always returns a result — either success or timeout error — before the server gives up. This prevents orphaned script executions.

## Server-Side State Tracking

The MCP server maintains a `tabMapping` for each plugin, updated by `tab.syncAll` and `tab.stateChanged` messages. This state is:

- Exposed to the side panel UI via the `config.getState` response, allowing the side panel to show plugin availability
- Used for diagnostics (`opentabs status`, `opentabs doctor`, `/health` endpoint)
- Cleared when the WebSocket disconnects (state is unknown until the extension reconnects and sends a fresh `tab.syncAll`)
