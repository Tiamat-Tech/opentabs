## Discovery

The MCP server discovers plugins from two sources on startup:

### Local Plugins

Paths listed in the `plugins` array of `~/.opentabs/config.json`:

```json
{
  "plugins": ["../plugins/slack", "/Users/me/my-plugin"]
}
```

Relative paths are resolved from the config directory. All paths must resolve under `$HOME` or `$TMPDIR` — symlinks are resolved via `realpath()` to prevent directory traversal.

### npm Plugins

The server scans `node_modules` for packages matching `opentabs-plugin-*` (scoped or unscoped) or with the `opentabs-plugin` keyword in `package.json`. Discovered packages are only loaded if listed in the `npmPlugins` allowlist:

```json
{
  "npmPlugins": ["opentabs-plugin-github", "@myorg/opentabs-plugin-jira"]
}
```

If the same plugin name appears in both local and npm sources, the local copy takes precedence.

## Plugin Metadata

Metadata lives in `package.json` under the `opentabs` field:

```json
{
  "name": "opentabs-plugin-slack",
  "version": "0.0.1",
  "opentabs": {
    "displayName": "Slack",
    "description": "Give AI agents access to Slack workspaces",
    "urlPatterns": ["*://*.slack.com/*"]
  }
}
```

### Validation

- **Plugin name**: lowercase alphanumeric with hyphens. Reserved names are rejected: `system`, `browser`, `opentabs`, `extension`, `config`, `plugin`, `tool`, `mcp`.
- **URL patterns**: Chrome match patterns. Overly broad patterns (`*://*/*`, `<all_urls>`, bare TLD wildcards) are rejected.
- **Tools**: at least one required. Descriptions capped at 1000 characters.
- **Adapter size**: maximum 5 MB.

## Adapter Injection

When a tab matching a plugin's URL patterns loads:

1. The extension detects the match and injects the adapter IIFE using file-based injection:

```typescript
chrome.scripting.executeScript({
  target: { tabId },
  world: "MAIN",
  files: [`adapters/${pluginName}.js`],
});
```

2. The IIFE executes and registers itself at `globalThis.__openTabs.adapters[pluginName]`

File-based injection bypasses page Content Security Policy — inline script injection would be blocked on sites like Slack and GitHub. Adapters run in the page's MAIN world with full access to DOM, cookies, localStorage, and fetch.

After injection, the extension verifies the adapter's `__adapterHash` against the expected hash. A mismatch triggers a retry; a persistent mismatch raises an error.

## File Watcher

The MCP server watches local plugin directories for changes to `dist/tools.json` and `dist/adapter.iife.js` (debounced at 200 ms).

- **`dist/tools.json` changed**: tool definitions are re-read, the MCP tool list is rebuilt, and the extension is notified via `plugin.update`
- **`dist/adapter.iife.js` changed**: the IIFE is re-read and the extension is notified

The watcher also monitors paths for plugins that failed initial discovery — when build artifacts appear, the plugin is loaded automatically. A fallback mtime-based poller runs every 30 seconds to catch stale `fs.watch()` instances on long-running processes.

## Tool Dispatch

When an AI agent calls a plugin tool:

1. MCP server receives the call at `/mcp` and validates input against JSON Schema
2. Server enforces a concurrency limit (max 5 concurrent dispatches per plugin)
3. Server sends a `tool.dispatch` JSON-RPC message to the extension
4. Extension finds the best matching tab by URL pattern and [tab ranking](/docs/concepts/tab-state-machine)
5. Extension runs the adapter's `handle()` in the tab's MAIN world
6. Result flows back through WebSocket to the MCP server and then to the AI agent
