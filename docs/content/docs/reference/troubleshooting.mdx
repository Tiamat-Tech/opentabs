## Extension not connecting

**Symptom**: The side panel shows "Disconnected" or `/health` reports `"extensionConnected": false`.

1. **Server running?** — `curl http://localhost:9515/health`. If it fails, start with `opentabs dev`.
2. **Correct port?** — Default is 9515. If using a custom port, make sure the extension is configured for the same one.
3. **Extension loaded?** — Open `chrome://extensions` and verify OpenTabs is loaded and enabled. Click any error link for details.
4. **Version mismatch?** — Run `opentabs doctor`. A mismatch between the installed extension and CLI version can cause failures. Rebuild with `bun run build` and reload the extension.
5. **Stale secret?** — The extension fetches the auth secret via `/ws-info` on first connect. If the secret changed, close and reopen the side panel to reconnect.

## Extension reconnection delays

**Symptom**: After the server restarts, the extension takes several seconds to reconnect.

The extension uses exponential backoff: 1s, 2s, 4s, 8s, up to a 30-second cap. Backoff resets after a successful connection. To force an immediate reconnect, close and reopen the side panel.

## Plugin showing as "unavailable"

**Symptom**: The plugin appears in the side panel but shows as "unavailable".

A plugin is unavailable when a matching tab exists but `isReady()` returns false — usually because the user is not logged in.

1. **Log in** — Sign into the target web app. The extension re-probes `isReady()` on navigation and tab events.
2. **Refresh the tab** — A fresh page load may be needed to detect the authentication state.
3. **Check isReady() logic** — Verify it checks the right signal (localStorage token, DOM element, cookie). The probe has a 5-second timeout.

## Plugin showing as "closed"

**Symptom**: The plugin shows as "closed" in the side panel or MCP tool list.

A plugin is closed when no Chrome tab matches its URL patterns.

1. **Open the target web app** — Navigate to a URL matching the plugin's `urlPatterns`.
2. **Check URL patterns** — Verify the `opentabs.urlPatterns` field in `package.json` uses valid [Chrome match pattern syntax](https://developer.chrome.com/docs/extensions/develop/concepts/match-patterns).
3. **Multiple windows** — The extension searches across all Chrome windows.

## Hot reload not working

**Symptom**: After `bun run build`, the server doesn't pick up changes. The `/health` endpoint's `reloadCount` stays the same.

Bun's file watcher can go stale on long-running processes (22+ hours).

1. **Check uptime** — Fetch `/health` and check the `uptime` field. If 22+ hours, restart the server.
2. **Restart**: `Ctrl+C` then `opentabs dev`
3. **Verify the build** — Ensure `bun run build` completed without errors. TypeScript errors mean no `.js` files changed.

## Extension changes not taking effect

**Symptom**: After modifying extension source and rebuilding, behavior is unchanged.

The Chrome extension does not auto-reload. After `bun run build`:

1. Open `chrome://extensions`
2. Find the OpenTabs card and click the circular refresh icon
3. Close and reopen the side panel

## Plugin changes not picked up

**Symptom**: After rebuilding a plugin, the server still serves old tool definitions.

The file watcher monitors `dist/tools.json` and `dist/adapter.iife.js`. If auto-detection fails:

1. **Trigger manually** — `opentabs reload`
2. **Check file watcher** — `curl -s http://localhost:9515/health | jq .fileWatcher`. A `pollDetections` count above 0 means the native watcher went stale.
3. **Verify the build** — Check that `dist/tools.json` and `dist/adapter.iife.js` exist.

## MCP client cannot connect

**Symptom**: Claude Code, Cursor, or another MCP client reports a connection or authentication error.

1. **Server running?** — `curl http://localhost:9515/health`
2. **Correct URL** — The endpoint is `http://localhost:9515/mcp` (not `/ws` or `/`)
3. **Auth token** — `Authorization: Bearer <secret>` must match the `secret` in `~/.opentabs/config.json`. Get it with `opentabs config show --json | jq -r .secret`
4. **Client config** — For Claude Code, verify your `.mcp.json` has `"type": "streamable-http"` and the correct URL and Authorization header
5. **CORS** — The server rejects requests with a browser `Origin` header. Native MCP clients don't send Origin headers, so this should only affect browser-based testing.

## Side panel not updating

**Symptom**: The side panel shows stale data (wrong plugins, old tab states).

1. **Close and reopen** — Right-click the extension icon and select "Open side panel"
2. **Reload the extension** — From `chrome://extensions`, click the refresh icon
3. **Check server connection** — If "Disconnected", see [Extension not connecting](#extension-not-connecting)
